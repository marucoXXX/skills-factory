# bdd-ingest-disclosure: 開示資料から仮説をアップデート

開示資料（有報・決算短信・統合報告書・コーポレートガバナンス報告書・サステナビリティレポート等）を読み込み、BDDプロジェクト状態を更新するスキル。

## 入力

1. **開示資料ファイル**: PDFが一般的（有報・決算短信は EDINET、統合報告書は会社HP）
2. **資料メタ情報**: 種類（有報/決算短信/統合報告書/CG報告書/サステナレポート）、対象期間、発行日
3. **bdd-projectディレクトリ**: 既存のプロジェクトディレクトリパス（デフォルト: `./bdd-project/`）

ファイル仕様は `bdd-core-issues/references/schema.md` を参照すること。

## 出力

このスキル実行後の変更:
- `sources/disclosure/` に開示資料ファイルが追加される
- `sources/index.json` に新ソースが登録される
- `facts.json` に開示由来のFactが大量に追加される（特に `quantitative` が多い）
- `hypotheses.json` の該当仮説が更新される（議事録ほど劇的な書き換えは少なく、補強が中心）
- `issues.json` に新規論点が追加される場合がある（リスク開示等から）

## 実行手順

### Step 1: プロジェクトの存在確認

`bdd-project/meta.json` が存在しない場合はエラー。`bdd-init` の実行を促す。

### Step 2: 開示資料の取り込み

1. ファイルを `sources/disclosure/S-DISC-NNN_{description}.{ext}` にコピー
2. `sources/index.json` に登録:
   ```json
   {
     "id": "S-DISC-001",
     "type": "disclosure",
     "subtype": "yuho | tanshin | integrated_report | cg_report | sustainability_report",
     "title": "X社 第50期有価証券報告書",
     "filename": "S-DISC-001_yuho_FY2024.pdf",
     "fiscal_period": "FY2024",
     "issued_at": "2025-06-15"
   }
   ```

### Step 3: 資料種別ごとに重点的に抽出する情報を判断

**有価証券報告書（yuho）の場合 - 最も網羅的**:
- 主要な経営指標等の推移（5年分のPL・BS主要項目）→ L2-07-01〜04
- 事業の状況・セグメント情報 → L1-03、L1-04
- 主要顧客・主要仕入先（売上比率10%超の取引先）→ L2-04-01、L2-05-01
- 従業員の状況 → L2-06-02
- 役員・株主の状況 → L2-06-01
- 設備の状況 → L2-05-03
- 事業等のリスク → L1-09すべて
- コーポレートガバナンスの状況 → L2-10-04
- 訴訟事件 → L2-09-03

**決算短信（tanshin）の場合 - 直近期の業績中心**:
- 直近四半期・通期の財務数値 → L1-07
- セグメント別業績 → L1-03、L1-08
- 業績予想 → L2-08-01

**統合報告書（integrated_report）の場合 - 戦略開示中心**:
- 中期経営計画の進捗 → L2-08-01
- マテリアリティ・サステナビリティ → L1-10
- 価値創造プロセス・ビジネスモデル説明 → L1-03
- KPI・非財務指標 → L1-06、L1-10
- 経営陣メッセージ → 戦略・方針の理解

**コーポレートガバナンス報告書（cg_report）**:
- 取締役会の構成・独立性 → L2-10-04
- 政策保有株式 → L2-09-04
- 内部統制 → L2-10-04

**サステナビリティレポート（sustainability_report）**:
- 環境データ（CO2・エネルギー・廃棄物）→ L2-10-02
- 社会データ（労務・人権・ダイバーシティ）→ L2-10-03
- ガバナンス → L2-10-04

### Step 4: 事実（Fact）を抽出

**数値情報の抽出（quantitative）**:
- 売上・利益・利益率の時系列値（過去5年分すべて取る）
- 従業員数、平均年齢、平均年収、平均勤続年数
- 株主構成（上位10名の議決権比率）
- 設備投資額、減価償却費、R&D費用
- 主要顧客の売上比率（10%超開示分）
- セグメント別売上・営業利益

**定性情報の抽出（qualitative）**:
- 主要顧客名（具体的な企業名）
- 主要仕入先名
- 取引先との契約条件特記事項
- 事業上のリスク要因の文言
- 訴訟事件の概要

**直接引用したい記述（quote）**:
- 中期経営計画の柱・目標
- 経営陣メッセージの重要文言
- リスク開示の特に重要な部分

各Factに付ける情報:
- `source_id`: このS-DISC-NNN
- `source_location`: ページ番号（必須。有報なら「p.45」のように）
- `linked_issues`: その情報が関連する論点ID（複数可）
- `type`: `quantitative` / `qualitative` / `quote`
- `value` / `unit`: 数値情報のとき必須

ID採番: `facts.json` の既存最大ID + 1から連番

**抽出の優先度（時間が限られている場合）**:
1. 必ず取る: 5年分のPL（売上・営業利益・経常利益・純利益）、セグメント別売上、主要顧客（10%超）、従業員数、株主構成上位10、事業リスク
2. できれば取る: BS主要項目、CF主要項目、設備投資、R&D、地域別売上、子会社一覧
3. 余力があれば: 役員報酬、研究開発の詳細、設備の詳細、サステナビリティKPI

### Step 5: 仮説のアップデート

抽出したFactを使って仮説を更新。基本的なロジックは `bdd-ingest-minutes` と同じ:

**ただし開示資料特有の論点**:

1. **数値で覆される仮説への対応**:
   - IM段階で「売上CAGR約10%」という仮説だったが、有報を見たら7%だった場合
   - → 既存 `current` を `history` に退避（`superseded_reason`: 「有報（S-DISC-001）の5期分PLから実績CAGR7%と判明」）
   - → 新 `current` を作成（`confidence: "High"`、`supporting_facts` に該当Fact ID）

2. **数値が揃って強化される仮説**:
   - 「収益性は業界平均並み」という仮説に対して、有報で営業利益率の時系列が確認できた
   - → `current.supporting_facts` に追加、`current.confidence` を上げる
   - → `history` には退避しない（仮説の中身が変わってないため）

3. **開示の「事業等のリスク」記載との整合**:
   - 既存のL1-09配下の仮説と、開示されたリスクが対応するか確認
   - 開示にあるが既存仮説に反映されていないリスクがあれば、仮説を更新するか新規論点として追加

**confidence判定ガイド（開示資料の場合）**:
- `High`: 監査済み開示資料の数値・複数年データで一貫している
- `Mid`: 開示にはあるが断片的、または1期分だけ
- `Low`: 開示の文言から推測した仮説

### Step 6: 新規論点の追加（必要な場合）

開示資料を読んでいて、既存`issues.json`にない重要論点が見つかった場合:

- 「事業等のリスク」に対象会社特有のリスクが開示されていて、L1-09の既存論点でカバーできない場合
  - 例: 「特定の特許切れリスク」「サイバーインシデントの過去履歴」
- 統合報告書のマテリアリティとして特殊な論点が挙げられている場合

新規L2論点を `issues.json` に追加し、対応する仮説も `hypotheses.json` に追加する。

### Step 7: 財務モデルとの紐付けトリガー

開示資料から財務数値が大量に入ったら、財務モデル構築のタイミング。
ユーザーに `bdd-financial-model` スキルの実行を提案する。

### Step 8: サマリーをユーザーに提示

実行後の報告内容:

1. **取り込んだ開示資料**: ソースID、種別、対象期間
2. **抽出されたFact数**: type別（quantitative中心になるはず）、論点別の分布
3. **更新された仮説**:
   - 強化された仮説（補強・confidenceアップ）の件数
   - 修正された仮説（数値による反証等）の件数と内容
4. **新規論点**: あれば件数と内容
5. **注目すべき発見**:
   - 想定と異なる財務実態（成長率・利益率・コスト構造の違い）
   - リスク開示で初めて見えた重要事項
   - 仮説間の矛盾（例: 「成長性高い」仮説と実際の鈍化）
6. **次のアクション提案**:
   - 財務モデル構築のタイミング判断
   - マネインで深掘りすべき論点

## 注意事項

- **数値の正確性を最優先**: 開示資料の数値は事実なので、正確に転記する。単位（百万円・千円）に注意
- **時系列の整合性**: 5期分の数値を取るなら必ず5期すべて取る。歯抜けは避ける
- **ページ番号の記録**: `source_location` にページ番号を必ず書く。後で参照できなければ Fact の信頼性が下がる
- **開示の限界を認識**: 開示資料は対象会社が選んで出しているもの。「開示されていない＝存在しない」ではない。Mid/Lowの仮説は残しておくべき
- **仮説のhistory管理を厳格に**: 数値で仮説が覆ることはBDDの本質。古い仮説を消さず、なぜ変わったかを記録することがプロジェクトの説明責任を担保する
