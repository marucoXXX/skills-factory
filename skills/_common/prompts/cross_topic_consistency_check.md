# 論点間整合性チェック（共通パターン）

> **このファイルは `skills/_common/prompts/cross_topic_consistency_check.md` です。**
> オーケストレータースキル（business-deepdive-agent / 将来の comparison-synthesis-agent 等）の SKILL.md からは、`<!-- source: skills/_common/prompts/cross_topic_consistency_check.md (manual sync until D2) -->` コメント付きで**手動コピペ**してください。
> このファイルを変更したら `grep -rn "source: skills/_common/prompts/cross_topic_consistency_check.md" skills/*/SKILL.md` で被参照スキルを全て検出し、コピペし直すこと（ISSUE-001 D2 で自動化検討中）。

複数の論点で 1 つのデッキを構成するオーケストレーターでは、各論点が独立して LLM に処理されるため、**論点間の意味的整合性**が崩れやすい。既存の品質ゲート（main_message ルール / fill_*.py validate / visual-quality-reviewer）はすべて**単一スライドの自己整合性**しか見ないため、論点間の混同は構造的に検出されない。

本ファイルは「データ収集 → JSON 組み立て → スライド生成」の流れの中で、論点間整合性を強制するチェック機構を共通化する。

## 絶対ルール

### ルール1: 全論点で扱う主語は同じ事業セグメント名

例: `business-deepdive-agent` で「施設運営事業」を対象とした場合、論点 1〜5 すべての主語は「施設運営事業」。論点ごとに「全社」「グループ」「他事業」に揺れない。

### ルール2: 「顧客市場」は**顧客が属する業種の市場**であり、自社の事業市場ではない

これは特に **B2B 受託業（自分自身がサービス提供者の場合）** で混同が起きやすい:

| 業態 | 自社の事業市場（NG） | 顧客市場（OK） |
|---|---|---|
| ビルメンテナンス業 | ✗ ビルメンテナンス市場 | ✓ 不動産業 / オフィスビル賃貸市場 |
| 半導体製造装置メーカー | ✗ 半導体製造装置市場 | ✓ 半導体メーカーの設備投資市場 |
| HR Tech SaaS | ✗ HR Tech 市場 | ✓ 企業の人事関連支出市場 |
| 建設コンサルタント | ✗ 建設コンサル市場 | ✓ 建設投資市場 / インフラ整備需要 |
| 物流アウトソース | ✗ 物流アウトソース市場 | ✓ 顧客業種（製造業 / 小売業 等）の出荷量・売上 |

判別基準: **「自社が消えても伸びる市場」が顧客市場**。自社の事業市場（受託金額の総和）は自社が属する受託業界の指標であり、論点 5 の対象ではない。

### ルール3: 業界バリューチェーン（差別化）は自社が属する業界

論点 3（差別化＝バリューチェーン上のポジショニング）で扱う業界は、**自社が属する業界**であり、顧客の業界ではない。論点 5（顧客市場）と業界が異なることに注意:

- ビルメンテ業の場合: 論点 3 = ビルメンテ業界バリューチェーン、論点 5 = 不動産業界市場
- 同じスライドで両方の業界を扱うことはない

## 整合性チェック表テンプレ（Step 3 ユーザー承認時に必須）

オーケストレーターの Step 3（ユーザーへの Markdown 提示・承認）で、各論点の主語を整理した整合性確認表を**必ず含める**こと。LLM が JSON を組み立てる前に明示的に点検する強制プロセス。

```markdown
## 論点間整合性確認（必須）

| # | 確認項目 | 期待される値 | 実態 | OK? |
|---|---|---|---|---|
| 1 | 全論点の主語 = 論点 1 の `segment_name` | <セグメント名> | <各論点の主語> | ✅/❌ |
| 2 | 論点 4 顧客の業種 = 論点 5 顧客市場の業種 | <顧客業種> | <論点 5 で扱う市場の業種> | ✅/❌ |
| 3 | 論点 3 業界 = 自社の業界 | <自社業界> | <論点 3 で扱う業界> | ✅/❌ |
```

❌ がある場合は JSON を組み立てる前に修正する。

## アンチパターン

- ❌ 論点を独立に処理して、各論点の主語が揺れる（全社/グループ/他事業）
- ❌ B2B 受託業で論点 5（顧客市場）に自社の事業市場を入れる（典型的な混同）
- ❌ Step 3 で個別論点の内容だけ確認して、論点間整合性を確認しない
- ❌ visual-quality-reviewer の役割を「論点間整合性検査」と誤解する（visual-quality-reviewer は単一スライドの視覚品質のみ）

## 失敗事例（参考）

**Phase 3 (二幸産業 施設運営事業)**: 論点 5 に「ビルメンテナンス市場」のデータを入れた。設計上は「顧客 = ビルオーナー法人 = 不動産業」が属する市場（不動産業 / オフィスビル賃貸市場）を扱うべきだったが、自社の事業市場（受託金額ベースのビルメンテ市場 4-5 兆円）を入れたため、論点 4（顧客 = 東京建物・不動産業）と論理的整合性が取れない状態に。

**教訓**: 論点 5 の検索クエリを実行する前に、論点 4 で確定した `customer_industry` を入力に取る順序を強制すれば構造的に防げる。
