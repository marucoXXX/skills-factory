# 事業モデル整理の良い例・悪い例

## テーマ：SaaS型HR管理プラットフォーム

### ✅ 良い例

**Main Message**
クラウドHR領域の二面性を活かし、データ連携基盤としてのポジションを確立すべき

**Chart Title**
HR SaaSプラットフォームの事業構造

**自社サービス**
- 名前：HRクラウド
- 説明：中堅企業向け人事管理SaaSプラットフォーム

**サプライヤー**
1. **AWSクラウド**（クラウドインフラ事業者）
   - → 自社へ：サーバー・ストレージ・CDN等のクラウドインフラ
   - ← 自社から：月額インフラ利用料（従量課金）
2. **給与計算エンジン社**（給与計算APIプロバイダー）
   - → 自社へ：給与計算・税務計算API
   - ← 自社から：API利用料（トランザクション課金）

**顧客**
1. **中堅企業人事部門**（従業員100〜1,000名規模）
   - ← 自社から：統合人事管理SaaSサービス（勤怠・評価・給与）
   - → 自社へ：月額サブスクリプション料（1人あたり500円/月）
2. **人材紹介エージェント**（採用支援会社）
   - ← 自社から：採用候補者マッチングAPI・求人掲載枠
   - → 自社へ：マッチング成約時の成功報酬（年収の15%）

**意味合い**
1. プラットフォーム効果：人事データの蓄積により、採用・評価・配置のAI最適化が可能になり、SaaSの付加価値が時間とともに増大する
2. 収益多層化：サブスクリプション収益に加え、人材紹介の成功報酬が高マージン収益源となっており、景気変動への耐性がある
3. サプライヤー依存リスク：クラウドインフラがAWS単一依存のため、価格交渉力が弱く、マルチクラウド対応が中期的な課題

---

### ❌ 悪い例（よくある間違い）

**Main Message の NG例**
- 「事業モデルを整理した」← アクションを促していない
- 「HRクラウドについて」← 「〜すべき」で終わっていない
- 「HRクラウドは良いサービスなので拡大すべき」← 具体性がない

**サプライヤー・顧客の NG例**
- サプライヤー：「ITベンダー」← 具体性がない。何を提供しているかが不明
- 顧客：「企業」← セグメントが不明確。どんな企業か、何を求めているか不明
- 取引フロー：「サービス提供」「お金を払う」← 具体性がない。何のサービスか、いくらかが不明

**意味合いの NG例**
- 「良いビジネスモデルである」← 評価だけで示唆がない
- 「リスクがある」← 何のリスクかが不明
- 「成長可能性がある」← 根拠と具体性が欠如

## テーマ：フードデリバリープラットフォーム

### ✅ 良い例

**Main Message**
三面マーケットプレイスの規模拡大に向け、配達員ネットワークの稼働率向上を最優先で取り組むべき

**Chart Title**
フードデリバリー事業の三面構造

**自社サービス**
- 名前：デリナウ
- 説明：飲食店とユーザーをつなぐフードデリバリープラットフォーム

**サプライヤー**
1. **飲食店パートナー**（加盟レストラン）
   - → 自社へ：調理済み食事の提供
   - ← 自社から：注文情報の送客・タブレット端末の無償提供
2. **配達パートナー**（個人事業主の配達員）
   - → 自社へ：配達サービスの提供
   - ← 自社から：配達1件あたりの報酬（基本料300円＋距離加算）
3. **決済プロバイダー**（クレジットカード会社）
   - → 自社へ：オンライン決済処理
   - ← 自社から：決済手数料（取引額の3.5%）

**顧客**
1. **個人ユーザー**（都市部の20〜40代）
   - ← 自社から：飲食店からの食事のデリバリー・リアルタイム追跡
   - → 自社へ：注文代金＋配送料（380円〜）＋サービス手数料（10%）

**意味合い**
1. ネットワーク効果：飲食店数の増加→ユーザー増加→配達員増加の好循環が成立しつつあるが、各面のバランス維持が成長の鍵
2. 配達員稼働率が収益性のボトルネック：ピーク時間帯以外の配達員稼働率が35%に留まっており、固定費的な配達報酬が利益率を圧迫
3. 飲食店のマルチホーミングリスク：加盟店の70%が競合プラットフォームにも出店しており、手数料率の引き上げ余地が限定的
