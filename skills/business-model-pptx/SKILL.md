# 事業モデル図 PowerPoint ジェネレーター

自社サービスを中心に、サプライヤー（仕入先）と顧客の取引関係を模式図として描画し、
戦略コンサルタント品質のPowerPointスライドを生成するスキル。

---

## 事業モデル図とは

事業モデル図は、企業の収益構造・取引構造を一目で理解できる模式図。以下の3つの主体と、その間の取引フローで構成される：

| 要素 | 配置 | 説明 |
|------|------|------|
| **サプライヤー** | 図の左側 | 自社に財・サービスを提供する主体（仕入先、パートナー、技術提供者など）。1〜4社程度 |
| **自社（中心）** | 図の中央 | 自社および自社のサービス・プロダクト名。太枠のボックスで強調表示 |
| **顧客** | 図の右側 | 自社から財・サービスを受ける主体（法人顧客、消費者、代理店など）。1〜4社程度 |
| **取引フロー（矢印）** | 主体間 | 各主体間で「何が流れるか」を矢印＋テキストで表現（サービス提供→、←対価支払い等） |

### スライド全体の構成

| 要素 | 定義 | ポイント |
|------|------|------|
| **Main Message** | スライド全体を総括し、読み手にアクションを促す一文。最大70文字 | 必ず「〜すべき」で締める |
| **Chart Title** | スライドのサブタイトル。事業モデルのテーマや文脈を端的に示す短いフレーズ（10〜20文字） | Main Messageを補足する文脈を示す |
| **事業モデル図** | 左のサプライヤー→中央の自社→右の顧客の取引フロー模式図 | HTMLで描画→スクリーンショット→PPTXに画像挿入 |
| **意味合い（Implications）** | 事業モデルから読み取れる示唆・ポイント。3点 | 事業の強み、リスク、成長機会など |

### 記述ルール

- **Main Message**: 最大70文字。事業モデルの要点を一言で総括し、必ず「〜すべき」で終える。ユーザーが指定した場合はそのまま使用、指定がない場合はドラフトして確認を取る。
- **Chart Title**: 10〜20文字程度。Main Messageのテーマを補足する文脈フレーズ。
- **意味合い**: 3点。事業モデルの特徴・強み・リスク・戦略的示唆を端的に記述。各項目は「ラベル：詳細の文章」形式。
- **サプライヤー・顧客**: それぞれ1〜4主体。各主体は名前と簡潔な説明を持つ。
- **取引フロー**: 各主体間に上向き矢印（提供）と下向き矢印（対価）の2本を基本とする。

### 図中テキストの文字数制限（重要）

事業モデル図はPowerPointのスライド内に画像として挿入されるため、テキストが長すぎると省略（...）されたりはみ出す。以下の文字数を厳守すること：

| 要素 | 最大文字数 | 例 |
|------|------|------|
| **サプライヤー/顧客の名前** | 10文字以内 | ○ AWSクラウド、× クラウドインフラストラクチャープロバイダー |
| **サプライヤー/顧客の説明** | 15文字以内 | ○ クラウドインフラ事業者、× 大手クラウドコンピューティングサービス |
| **取引フローのテキスト** | 15文字以内 | ○ 給与計算API提供、× 給与計算・税務計算APIサービスの提供 |
| **自社サービスの名前** | 12文字以内 | ○ HRクラウド |
| **自社サービスの説明** | 20文字以内 | ○ 人事管理SaaSプラットフォーム |

テキストが長い場合は、要約して短くすること。括弧内の補足情報は省く、または意味合い側に記載する。

## 入力パターンと処理フロー

### パターンA：議事録・文字起こしが入力された場合

**いきなりPowerPointを作成しない**

1. **Step 1: 事業モデル情報をドラフトする**（後述の抽出ガイドラインに従う）
2. **Step 2: Markdownでユーザーに提示**し、確認・修正を求める
3. **Step 3: ユーザーの承認後**、PowerPointを生成する

### パターンB：事業モデルの詳細が入力された場合

1. **Step 1: 内容を確認**し、主体・取引フローが揃っているか確認
2. 不足・曖昧な点があれば修正提案を行う
3. **Step 2: 確認後、PowerPointを生成する**

---

## Step 1: 議事録から事業モデル情報を抽出する

### 自社サービスの特定

議事録から**自社の中核サービス・プロダクト名**を特定する。以下のような表現を探す：
- 「当社の〜」「弊社のサービスは〜」「我々のプラットフォーム」
- サービス名・ブランド名の直接的な言及
- 「〜を提供している」「〜事業を展開」

### サプライヤーの特定

自社に財・サービスを提供する主体を特定する：
- 「〜から仕入れている」「〜と提携している」「〜がインフラを提供」
- 原材料メーカー、クラウド事業者、外部ベンダー、ライセンス元
- 各サプライヤーごとに「何を提供しているか」「何を対価として受け取っているか」を整理

### 顧客の特定

自社がサービスを提供する相手を特定する：
- 「〜に販売している」「ターゲット顧客は〜」「エンドユーザーは〜」
- 法人顧客、消費者、代理店、パートナー
- 各顧客ごとに「何を提供しているか」「何を対価として受け取っているか」を整理

### 意味合いの導出

事業モデルの特徴から戦略的な示唆を導く：
- このモデルの強み（差別化要因、ネットワーク効果、スケーラビリティ等）
- このモデルのリスク（サプライヤー依存、顧客集中、参入障壁等）
- 成長・改善の機会

## Step 2: 事業モデルのMarkdown出力フォーマット

ユーザーに確認を求める際は、以下のフォーマットで出力する：

```markdown
## 事業モデル整理結果

**Main Message（※ドラフト）**
〜すべき（最大70文字）

**Chart Title（※ドラフト）**
〜のテーマ

### 自社サービス
- 名前：XYZプラットフォーム
- 説明：〜を提供するSaaSサービス

### サプライヤー
1. **サプライヤーA**（原材料メーカー）
   - → 自社へ：原材料の供給
   - ← 自社から：仕入代金の支払い
2. **サプライヤーB**（クラウド事業者）
   - → 自社へ：クラウドインフラ
   - ← 自社から：月額利用料

### 顧客
1. **法人顧客**（大手製造業）
   - ← 自社から：SaaSサービスの提供
   - → 自社へ：月額利用料
2. **個人ユーザー**（消費者）
   - ← 自社から：無料プランの提供
   - → 自社へ：広告収入（間接）

### 意味合い（Implications）
1. プラットフォーム効果：〜
2. サプライヤーリスク：〜
3. 成長ポテンシャル：〜
```

確認メッセージ例：
> 上記の事業モデル整理でよろしいでしょうか？Main MessageとChart Titleも含めて修正があればお知らせください。確認後にPowerPointを生成します。

## Step 3: PowerPointの生成

### テンプレートの参照

テンプレートは `assets/business-model-template.pptx` を使用する。

```bash
TEMPLATE="<SKILL_DIR>/assets/business-model-template.pptx"
```

※ `<SKILL_DIR>` は実際のスキルインストールパスに置き換えること。

テンプレートのShape構造は `references/template-mapping.md` を参照。

### 事業モデルデータのJSON化

事業モデル情報を `{{WORK_DIR}}/business_model_data.json` に以下の形式で保存する：

```json
{
  "main_message": "〜すべき（最大70文字）",
  "chart_title": "〜のテーマ（10〜20文字）",
  "company": {
    "name": "自社サービス名",
    "description": "サービスの簡潔な説明"
  },
  "suppliers": [
    {
      "name": "サプライヤー名",
      "description": "簡潔な説明",
      "flow_to_company": "自社へ提供するもの（例：原材料の供給）",
      "flow_from_company": "自社から受け取るもの（例：仕入代金）"
    }
  ],
  "customers": [
    {
      "name": "顧客名",
      "description": "簡潔な説明",
      "flow_from_company": "自社から提供するもの（例：SaaSサービス）",
      "flow_to_company": "自社が受け取るもの（例：月額利用料）"
    }
  ],
  "implications": [
    {"label": "ラベルA", "detail": "詳細の文章"},
    {"label": "ラベルB", "detail": "詳細の文章"},
    {"label": "ラベルC", "detail": "詳細の文章"}
  ]
}
```

### スクリプト実行コマンド

#### 多事業モデル（businesses配列あり）の場合

自社が複数の事業ラインを持ち、各事業ラインが個別にサプライヤー・顧客と接続されるモデル。
`businesses` 配列が存在する場合、自動的にこちらのレイアウトが使われる。

```json
{
  "main_message": "〜すべき（最大70文字）",
  "chart_title": "〜のテーマ（10〜20文字）",
  "company": {
    "name": "自社名"
  },
  "businesses": [
    {"name": "事業ライン1", "description": "概要"},
    {"name": "事業ライン2", "description": "概要"},
    {"name": "事業ライン3", "description": "概要"}
  ],
  "suppliers": [
    {
      "name": "サプライヤー名",
      "description": "簡潔な説明",
      "connections": [
        {"to_business": 0, "label_to": "提供するもの", "label_from": "対価"},
        {"to_business": 1, "label_to": "提供するもの", "label_from": "対価"}
      ]
    }
  ],
  "customers": [
    {
      "name": "顧客名",
      "description": "簡潔な説明",
      "connections": [
        {"from_business": 0, "label_to": "提供するもの", "label_from": "対価"},
        {"from_business": 2, "label_to": "提供するもの", "label_from": "対価"}
      ]
    }
  ],
  "implications": [
    {"label": "ラベルA", "detail": "詳細の文章"},
    {"label": "ラベルB", "detail": "詳細の文章"},
    {"label": "ラベルC", "detail": "詳細の文章"}
  ]
}
```

**connections の仕様:**
- `to_business` / `from_business`: 接続先の事業ラインのインデックス（0始まり）
- `label_to`: 右向き矢印のラベル（サービス提供など。赤色で表示）
- `label_from`: 左向き矢印のラベル（対価支払いなど。青色で表示）。不要なら空文字列
- サプライヤーは `to_business` で接続先を指定、顧客は `from_business` で接続元を指定
- 1つのサプライヤー/顧客が複数の事業ラインに接続可能（行をまたがって配置される）

### スクリプト実行コマンド

```bash
apt-get install -y -qq fonts-noto-cjk 2>/dev/null
pip install python-pptx playwright -q --break-system-packages
playwright install chromium 2>/dev/null

python <SKILL_DIR>/scripts/fill_business_model.py \
  --data {{WORK_DIR}}/business_model_data.json \
  --template <SKILL_DIR>/assets/business-model-template.pptx \
  --output {{OUTPUT_DIR}}/BusinessModel_output.pptx
```

※ `<SKILL_DIR>` は実際のスキルインストールパスに置き換えること。

### 出力確認

```bash
python -m markitdown {{OUTPUT_DIR}}/BusinessModel_output.pptx
```

内容が正しく反映されているか確認し、ユーザーに提示する。

## 品質チェックリスト

PowerPoint生成後、以下を確認：

- [ ] Main Messageが70文字以内で「〜すべき」で終わっているか
- [ ] Chart Titleが10〜20文字程度で文脈を的確に示しているか
- [ ] 自社サービスが中央に配置されているか
- [ ] サプライヤーが左側、顧客が右側に配置されているか
- [ ] 各主体間の取引フロー（提供・対価）が矢印で表現されているか
- [ ] 意味合いが3点あり、各項目が「ラベル：詳細」形式になっているか
- [ ] PPTXのmarkitdown出力でプレースホルダーが残っていないか

## オーケストレーター連携

### `business-deepdive-agent` から呼び出される場合の規約

| 項目 | 値 |
|---|---|
| 入力 JSON ファイル名 | `data_NN_business_model.json`（NN は global_slide_offset 経由で親が採番） |
| 出力 PPTX ファイル名 | `slide_NN_business_model.pptx`（同上） |
| 入力ディレクトリ | `{{WORK_DIR}}/company-deepdive-agent/<parent_run_id>/segments/<segment_slug>/` |
| 出力ディレクトリ | 同上 |

`business-deepdive-agent` は本スキルを **5 論点中 2 番目（ビジネスモデルは？）** として呼び出す。
作業ディレクトリは `company-deepdive-agent` 配下のセグメント別 subdir に統一し、merge は親が担当。

単独起動時は `{{WORK_DIR}}/business_model_data.json` を使う既存の運用に従う（本セクションの規約は適用されない）。
