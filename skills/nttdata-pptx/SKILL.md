# NTT DATA ブランド PowerPoint スキル

このスキルは、NTT DATAグループの公式テンプレート（`assets/template.pptx`）をベースに、ブランドガイドライン準拠のプレゼンテーション資料を作成する。

---

## ポジショニング：戦略コンサルタントの成果物

NTT DATAテンプレートで作成するスライドは、戦略コンサルタントがクライアントに提出する成果物である。「見栄えが良い」だけでなく、「コンサルタントがいなくても内容が伝わる」自立したドキュメントを目指す。

---

## pptxスキルとの関係（上書きルール）

本スキルは既存のpptxスキルと併用する。技術的なワークフロー（unpack/pack、editing手順、QA手順、画像変換）はpptxスキルの `editing.md` に従うこと。デザイン・レイアウト・ビジュアル要素の選択はpptxスキルの `SKILL.md`「Design Ideas」セクションに従うこと。ただし、以下のルールで上書きする：

| pptxスキルの項目 | 本スキルでの扱い |
|---|---|
| Color Palettes | **上書き**: Accent1（#070F26）のみ使用。カラーパレット表は無視 |
| Typography（フォント） | **上書き**: Arial / Meiryo UI のみ。フォントペアリング表は無視 |
| Typography（サイズ） | **上書き**: プレースホルダー内はマスター任せ。追加要素は余白を埋める大きさにする |
| Spacing「Leave breathing room」 | **上書き**: 余白は悪。フォント・図を拡大して埋める（後述「テキスト密度ルール」参照） |
| 「Every slide needs a visual element」 | **補足**: ビジュアルは使うが、テキスト密度も重視。ビジュアルのためにテキストを削らない |

上書きされていない項目（レイアウト選択、Data display、QA手順等）はpptxスキルにそのまま従う。

## フォント

| 用途 | 欧文 | 日本語 |
|------|------|--------|
| 見出し | Arial | Meiryo UI |
| 本文 | Arial | Meiryo UI |

**フォントサイズはプレースホルダー内では指定しない。** `<a:rPr>` に `sz` を書くとマスターのサイズが上書きされる。レイアウトのデフォルトに任せること。プレースホルダー外に追加する `<p:sp>` や `<p:graphicFrame>` では、余白を埋めるために大きめのサイズを積極的に使う。

## プレースホルダー構造

`add_slide.py unpacked/ slideLayout1.xml` で作成されるコンテンツスライドには、以下の3つのプレースホルダーが含まれる。**これらを削除したり新しいシェイプで置き換えてはならない。テキストだけを編集すること。**

| プレースホルダー | 役割 | 編集方法 |
|---|---|---|
| `ph type="title"` | スライドタイトル | `<a:t>` のテキストだけを書き換える |
| `ph type="body" idx="12"` ★削除禁止 | キーメッセージ（青い太字） | `<a:t>` のテキストだけを書き換える |
| `ph idx="13"` | コンテンツエリア（本文） | テキストを記述。ビジュアル要素は別途 `<p:spTree>` に追加 |

## キーメッセージの原則 ★最優先

キーメッセージはスライド作成の出発点であり、最も重要な要素となる。

### ルール1: キーメッセージ = 完結した一文で書く

体言止めやダッシュ区切りは使わず、「〜となる」「〜である」「〜すべき」のように文として完結させる。

❌「SaaS市場は2028年に2.8兆円規模 — 今が参入の最適タイミング」
❌「市場分析の結果」
✅「SaaS市場は2028年に2.8兆円規模であり、今が参入の最適タイミングとなる」
✅「既存のRPA導入プロセスでは属人化が避けられないため、形式知化の仕組みを導入すべき」

### ルール2: キーメッセージ = 新しい発見 or 推奨アクション

タイトルの言い換えや単なる要約ではなく、そのスライドで伝えるべき主張・発見・提言を書く。

❌「本章では市場環境について分析する」（ただの章説明）
✅「国内SaaS市場は年率15%で成長しており、2028年に2.8兆円規模となる」

### ルール3: キーメッセージだけで全体のストーリーが読める

全スライドのキーメッセージを順番に読んだとき、提案の全体像が伝わるストーリーになっていること。構成提案時（フェーズ1）でキーメッセージの一貫性を検証する。

### ルール4: キーメッセージは最初に決める

スライドの構成要素を考える順序:
1. キーメッセージ（何を主張するか）
2. それを裏付けるコンテンツ（データ、根拠）
3. レイアウト（内容に合った見せ方）

## テキスト密度ルール

### ルール1: 余白は悪

フォントサイズや図のサイズを拡大して、スライド全体を埋める。「もう少し大きくできないか？」を常に問う。

### ルール2: 1行あたり約40文字

40文字を超える場合は箇条書きや改行で分割する。クライアントがコンサルタントの説明なしに自力で読めるだけのテキスト量を確保する。

### ルール3: 箇条書きは具体的に

各項目は2〜3行の説明を含める（1行で終わらせない）。抽象的なキーワードの羅列ではなく、具体的な内容を書く。

---

## プレゼン構成ルール

### ルール1: スライド構成

すべてのページの上部に必ず「伝えるべきキーメッセージ」と「スライドタイトル」を記載する。
コンテンツはスライドの中央部に作成すること。

### ルール2: 抽象と具体のメリハリ

抽象的なページ（ビジョン・方針）はテキスト少なめで余白を活かす。具体的なページ（分析・根拠）はデータと数字をしっかり書く。

## ワークフロー

### フェーズ1: 構成提案（生成前に必ず実施）

ユーザーから与えられたコンテキスト（テーマ、目的、素材など）をもとに、**スライド構成案をページ単位でユーザーに提示する。** ユーザーの承認を得てから生成を開始すること。

提案フォーマット例:
```
【スライド構成案】
1. 表紙 — タイトル: 〇〇〇
2. 背景と目的 — キーメッセージ: 〇〇〇 / レイアウト: 2カラム
3. 市場分析 — キーメッセージ: 〇〇〇 / レイアウト: 大数字コールアウト
4. 競合比較 — キーメッセージ: 〇〇〇 / レイアウト: 表
5. 導入ステップ — キーメッセージ: 〇〇〇 / レイアウト: プロセスフロー
6. まとめ — キーメッセージ: 〇〇〇
```

提案時のポイント:
- 各スライドのタイトル、キーメッセージ、レイアウト種類を明示する
- **キーメッセージを順番に読んで、ストーリーとして一貫しているか検証する**（ルール3）
- レイアウトはpptxスキルのDesign Ideas「Layout options」「Data display」から選ぶ
- スライド順序は内容に応じて柔軟に決める（固定順序はない）
- ユーザーが修正を求めたら構成を調整し、再度承認を得る

### フェーズ2: 生成

ユーザーの承認後、以下の手順で生成する。

1. `cp assets/template.pptx ./working.pptx`
2. `python scripts/thumbnail.py working.pptx` で分析
3. `python scripts/office/unpack.py working.pptx unpacked/`
4. 構造変更（不要スライド削除 → add_slide → 順序調整）
5. **構造変更を全て完了してから**コンテンツ編集
6. `python scripts/clean.py unpacked/`
7. `python scripts/office/pack.py unpacked/ output.pptx --original working.pptx`

**pack.py のバリデーションは必ず通すこと。** エラーが出た場合はXMLを修正して再実行する。手動zip（`zipfile` 等）によるパックやバリデーション無視は禁止 — ファイルが壊れる原因になる。サブエージェントにXML編集を任せる場合は、タグの閉じ忘れや属性の不正な位置（例: `<a:solidFill>` が `<a:rPr>` の外に出る）などの構造ミスが起きやすいため、pack.py のバリデーションが特に重要となる。

## 避けるべきこと

- Accent1以外の色を使う
- 角丸四角（ROUNDED_RECTANGLE）を使う
- すべてのスライドを同じレイアウトにする
- テキストだけのスライドを作る（pptxスキルのDesign Ideasに従い、必ずビジュアル要素を入れる）
- プレースホルダーを削除して新しいシェイプで置き換える
- `<a:rPr>` にフォントサイズや色を直接指定する（プレースホルダー内）
- idx="12" のキーメッセージプレースホルダーを削除する
- 箇条書きの各項目を1行で済ませる（具体的ページでは2〜3行の説明を書く）
- キーメッセージを体言止めやタイトルの言い換えにする
- スライドに余白を残す（フォント・図を拡大して埋める）
- フッターの著作権表記やNTT DATAロゴを削除する
